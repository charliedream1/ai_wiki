## 检索方案分析

这个仓库使用的是 **基于LLM的上下文学习（In-Context Learning）方案**，而不是传统的向量检索或关键词匹配。具体来说：

### 🎯 核心检索机制

**1. 全量上下文注入（In-Context Retrieval）**
- 在 `app.py` 的 `_format_background_info()` 方法（第480-493行）中，将所有FAQ问题格式化为背景信息
- 每次查询时，把**所有存储的Q&A**直接放入prompt中
- 通过LLM的理解能力来匹配相关问题

```python
def _format_background_info(self) -> str:
    if not self._questions:
        return "DATA NOT AVAILABLE"
    
    return "\n\n".join([
        f"""\
Question #{q.id}[variants={q.variants}][[
Answer: {q.answer}
]]
"""
        for q in self._questions.values()
    ])
```


**2. LLM智能匹配**
- 在 `ask_question()` 方法（第304-478行）中，构造详细的prompt
- Prompt包含：
  - 系统指令：要求仅基于背景信息回答
  - 所有FAQ作为背景信息
  - 用户问题
  - 要求LLM输出结构化的JSON结果

**3. 多阶段推理流程**
LLM被要求执行以下步骤：
- 理解用户问题的意图
- 从背景信息中找到相关的question variants
- 提取相关引用（quotes）
- 判断是否能完全/部分/无法回答
- 生成答案并自我批判
- 提取实体信息

### 📊 方案特点

**优点：**
- ✅ **语义理解强**：利用LLM的深度语义理解，能处理变体问题
- ✅ **可追溯性**：返回具体的question_id和引用片段
- ✅ **防幻觉**：严格限制只能使用提供的背景信息
- ✅ **简单直接**：无需向量数据库、embedding模型等额外组件

**缺点：**
- ❌ **扩展性差**：所有FAQ都放入context，受限于LLM的context window
- ❌ **成本较高**：每次查询都需要处理全部FAQ数据
- ❌ **性能问题**：FAQ数量增多时，token消耗和响应时间会显著增加

### 🔧 数据存储

使用简单的文档数据库存储：
- 开发环境：`JSONFileDocumentDatabase`（持久化到JSON文件）
- 测试环境：`TransientDocumentDatabase`（内存存储）

每个Question包含：
- `id`: 唯一标识
- `variants`: 问题的多个变体形式
- `answer`: 答案内容

### 💡 总结

这是一个**适合小规模FAQ场景**的检索方案，利用了LLM强大的语义理解能力来做"软匹配"，避免了传统检索系统需要的复杂基础设施。但当FAQ数量增长到一定程度（比如几百上千条），就需要考虑引入向量检索或其他优化方案了。
